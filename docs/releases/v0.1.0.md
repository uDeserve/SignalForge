# SignalForge v0.1.0

SignalForge v0.1.0 is the first public product-shaped release of the project.

This release establishes SignalForge as an AI-native feedback-to-issue harness for small web teams and independent developers that want a GitHub-native path from noisy user reality to decision-ready engineering work.

## Highlights

- aggregation-aware case formation instead of one-submission-one-issue behavior
- GitHub App publication flow with webhook-driven maintainer decision sync
- adapter-first integration path for existing web applications
- repo-local CLI for fast setup with `sf:init`, `sf:doctor`, and `sf:start`
- coding-agent-friendly integration assets and scaffold outputs
- stronger open source trust signals including license, changelog, support, and security policy

## Why This Release Matters

Modern automation can already operate on a good GitHub issue.

SignalForge focuses on the missing upstream layer:

```text
user feedback + runtime pain
-> SignalForge aggregation and synthesis
-> decision-ready GitHub issue
-> maintainer or agent execution
```

This release makes that category legible in both product positioning and repository structure.

## What Is Ready

- feedback intake through API and adapter paths
- aggregation-aware triage and case synthesis
- GitHub issue publication
- GitHub App auth path
- webhook-based decision sync from GitHub comments
- repo-local setup and verification commands
- agent-oriented setup contracts and integration documentation

## Validation

SignalForge has already been validated in a real deployment flow:

- live HTTPS deployment
- real GitHub App issue publication
- real webhook delivery
- real maintainer decision sync back into SignalForge state

See `docs/live-e2e.md` for the verification record.

## Intended Users

- small web application teams
- independent developers
- AI-native product teams that want to route user reality into GitHub-native execution loops

## Known Gaps

- no polished hosted product path yet
- runtime monitoring integrations still need first-class bridges
- delegated execution and PR automation are not yet the primary shipped loop
- operator observability still needs more depth
